# Plan 008: System Prompt Optimization + Memory Seeding

## STOP: Read This Entire Document Before Making Any Changes

This plan reduces Edward's per-turn token cost by ~40-45% on tool descriptions and introduces memory seeding — a pattern where Edward starts with experience-like instruction memories that teach him WHEN and WHY to use capabilities proactively. A light AUTONOMY_FRAMEWORK enhancement rounds it out.

**Dependencies**: All prior plans (001-007) completed
**Estimated effort**: 1 day

---

## Context & Rationale

### The Problem: Tool Descriptions Are Too Heavy

Edward's system prompt includes verbose tool description sections (generated by `get_*_tools_description()` functions) that duplicate information already in tool schemas (docstrings). When many skills are enabled, these add **~3,800-6,000 tokens per turn** — all spent describing WHAT tools do and HOW to call them, information that's already in the tool JSON schema.

| Section | Current ~Tokens | What's Redundant |
|---------|----------------|-----------------|
| NotebookLM | ~650 | 28-tool numbered catalog with parameters |
| Plan Tools | ~675 | When-to/when-not-to lists, detailed examples |
| Scheduled Events | ~525 | Parameter listings, column-level specifics |
| SQL/Databases | ~500 | Full medication tracking example |
| Orchestrator | ~350 | Verbose best practices |
| Others combined | ~1,300 | Parameter lists, return value descriptions |
| **Total** | **~4,000** | |

### The Problem: Edward is Reactive With Powerful Tools

Edward has 60+ tools including NotebookLM (28 tools), web search, persistent databases, scheduled events, and artifact generation. But the system prompt only tells him WHAT these tools do — not WHEN to proactively use them. The single proactive line is: *"Build knowledge proactively — if a topic comes up repeatedly, create a notebook for it."*

### The Solution: Trim + Seed

1. **Trim tool descriptions** to "when/why" guidance. Remove "what/how" that tool schemas already cover. Target: ~2,000 tokens saved per turn.
2. **Seed instruction memories** that encode behavioral patterns Edward has "learned." These surface via vector similarity only when contextually relevant — zero prompt cost on irrelevant turns.
3. **Enhance AUTONOMY_FRAMEWORK** with two small additions (~25 tokens, cached).

### Why Memory Seeding Over More System Prompt

| Approach | Per-Turn Cost | Contextual? | Evolvable? |
|----------|--------------|-------------|------------|
| System prompt rules | Every turn | No | No (hardcoded) |
| Seeded memories | Only when relevant | Yes (vector similarity) | Yes (Edward can update/forget) |

Memory seeding aligns with Edward's "values, not rules" philosophy. The memories feel like experience, not instructions. They compete equally with user-derived memories in retrieval, and Edward can modify them as he develops better patterns.

## Phase 3: Memory Seed Script

### Step 3.1: Create Script Directory

Create `backend/scripts/__init__.py` (empty file).

### Step 3.2: Create Seed Script

**New file**: `backend/scripts/seed_memories.py`

The script:
1. Initializes the database connection via `init_db()`
2. Defines 16 seed memories as a list of dicts
3. For each seed, runs `retrieve_memories(content, limit=1)` to check for similar existing memories
4. If top match similarity >= 0.80, skips (already exists)
5. Otherwise, stores via `store_memory()` with these fields:
   - `memory_type="instruction"` — behavioral guidance
   - `temporal_nature="timeless"` — never decays
   - `tier="knowledge"` — 1.2x retrieval boost
   - `reinforcement_count=5` — matches knowledge tier threshold
   - `importance=0.80-0.90` — high but not maximum

#### How to Run

**Prerequisites**: Backend database must be running (Docker PostgreSQL).

**From the backend directory** (with venv activated):

```powershell
# Windows (PowerShell)
cd c:\Users\cchen362\OneDrive\Desktop\meet-edweird\backend
.venv\Scripts\Activate.ps1
python -m scripts.seed_memories
```

```bash
# macOS/Linux
cd backend
source .venv/bin/activate
python -m scripts.seed_memories
```

The script will:
- Print each memory: either `CREATED (id=xxx)` or `SKIP (similar exists, score=0.xx)`
- At the end: `Done: X created, Y skipped`

**Idempotent**: Safe to run multiple times. Re-running after initial seed will skip all 16.

## Phase 4: Memory Audit (Optional, Manual)

**This phase is optional and can be done before, after, or independently of the code changes.** The seed script has duplicate detection, so it's safe to run regardless of existing memory state.

### Why Audit

Over time, Edward accumulates memories through conversation. Some may be:
- **Duplicates**: Same fact phrased differently (consolidation may not have caught all)
- **Outdated**: Old locations, stale preferences, completed projects
- **Low quality**: Vague or unhelpful ("User mentioned something about cooking")
- **Mistyped**: Should be a document instead of a memory

### How to Audit

**Step 1**: Export all memories (backend must be running at http://localhost:8000):

```powershell
# Get all memories (paginate if >100)
curl http://localhost:8000/api/memories?limit=100 | python -m json.tool > memories_export.json

# Or filter by type
curl "http://localhost:8000/api/memories?memory_type=instruction&limit=100"
curl "http://localhost:8000/api/memories?memory_type=fact&limit=100"
curl "http://localhost:8000/api/memories?memory_type=context&limit=100"
```

**Step 2**: Review the export. Look for:
- Duplicate content (check `content` field)
- Low importance scores that should be higher (or vice versa)
- `temporal_nature="timeless"` on things that are actually temporary
- Memories that would be better as documents (long text, recipes, instructions)

**Step 3**: Clean up via API:

```powershell
# Delete a bad memory
curl -X DELETE http://localhost:8000/api/memories/{memory_id}

# Update a memory's content, type, or importance
curl -X PATCH http://localhost:8000/api/memories/{memory_id} -H "Content-Type: application/json" -d '{\"content\": \"Updated content\", \"importance\": 0.8}'
```

**Step 4**: Run the seed script (Phase 3).

**Step 5**: Verify seeds were created:

```powershell
curl "http://localhost:8000/api/memories?memory_type=instruction&limit=50"
```

You should see 16 instruction-type memories with `tier: "knowledge"`.
